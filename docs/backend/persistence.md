# Persistence Foundation

BoardTrace uses PostgreSQL through SQLAlchemy 2's async engine and `async_sessionmaker`. The engine is constructed by the app factory but never connects during import. Readiness executes `SELECT 1`; liveness remains application-only.

Models use typed declarative mappings, PostgreSQL UUID primary keys, timezone-aware timestamps, and deterministic metadata naming conventions. Native PostgreSQL enums are selected for database-level type safety. They are versioned explicitly in Alembic migrations; adding enum values requires a reviewed migration. Alembic retains its empty `alembic_version` table after a downgrade to base; all application tables and enum types are removed.

`User -> Game -> GameFrame/Position` uses database cascades to delete capture references when a parent game is deleted. Frames contain only a scoped object-storage key and metadata: no image binary is stored. Model and engine artifacts similarly remain outside PostgreSQL; JSONB metadata/configuration uses callable `dict` defaults.

Repositories are domain-specific and never commit. A request receives one session; service/use-case code owns transaction boundaries using `transaction_scope`. No client-facing schema shares a persistence model, and no API exposes `EngineAnalysis` fields.

Alembic owns all schema changes. Use `pnpm db:upgrade`, `pnpm db:downgrade`, and `pnpm db:migrate`; review autogenerated revisions before applying. Integration and migration tests require `BOARDTRACE_TEST_DATABASE_URL` pointing at a separate PostgreSQL database and are marked `integration`, `database`, and/or `migration`; unit tests do not use SQLite.

Analysis orchestration uses a persistent `analysis_jobs` record and a transactional
`analysis_job_outbox` event in the same PostgreSQL transaction as completed-game ingestion.
The outbox is delivered at least once through a narrow queue port; consumers use row locks,
status checks, worker-owned leases, and terminal-state deduplication. Operational job and
outbox records contain identifiers and bounded error codes only. They are deleted by the
existing game/user cascades; a time-based operations retention policy remains deployment work.

Celery uses Redis only as a JSON broker on the explicit `boardtrace.analysis.jobs` queue.
Messages contain only a schema version, job ID, and correlation ID. The Prompt 9 handler is a
deterministic orchestration no-op: successful job execution does not set game analysis
availability and creates no engine result.

Prompt 10 adds an internal, typed `StockfishEngine` UCI adapter. It launches a configured
native executable only from a backend worker call after server-side completion verification;
imports and the FastAPI application never launch it. This foundation does not wire Stockfish
into the queue handler, persist output, or add a public response schema.
